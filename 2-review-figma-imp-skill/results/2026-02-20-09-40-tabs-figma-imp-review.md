### Subcomponent classification note

Per the updated additional rules, subcomponent classification should use the **component definition name** (the master component the instance references), not the layer/instance name. The Figma MCP tools (`get_variable_defs`, `get_design_context`, `get_metadata`) do not expose a `mainComponent.name` or `componentName` field for nested instances. The `get_metadata` call on variant node 3672:743 ("Alignment=Left, Tabs=8") returned only the symbol-level entry without expanding nested instances.

**Subcomponent classification could not be applied by component definition name.** The code generated by `get_design_context` uses the name "Tab" for the nested component. This was used as a proxy for the component definition name. Since "Tab" does not start with '.' or '_', it is treated as **not a true subcomponent** (override-only scope). However, if the actual Figma component definition name begins with '.' or '_' (e.g. ".Tab"), the Tab should instead be treated as a full subcomponent with all checks applied. This should be verified manually in Figma or via a component export that includes `mainComponent.name`.

### Analysis

The MCP `get_variable_defs` was called on the Tabs component set (node 3672:742). The component set exposes **Alignment** (Left, Center) and **Tabs** (2–8) as top-level variant properties. Interaction states (Hover, Pressed, Focus-Visible, Disabled) are managed within the nested Tab component's variants.

The `get_variable_defs` response includes variables from the entire node subtree, but only for the variants present in the component set. Since the Tabs component set displays Tab instances in their **Resting** interaction state, only resting-state tokens appear. The 27 missing tokens correspond to non-resting interaction variants of the Tab component.

**Subcomponent scope:** Since the Tab component's definition name could not be verified via MCP (see classification note above), and the proxy name "Tab" does not start with '.' or '_', Tab internals are treated as out of scope (override-only). Under this classification, the 27 missing tokens would not be reportable as in-scope issues. **However, if the Tab component's definition name starts with '.' or '_', these 27 tokens become fully in scope and should be verified by querying the Tab component node directly.**

No position tokens were present in the CSV, so no exclusions per the position token rule were needed.
